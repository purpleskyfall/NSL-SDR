# B1I signal simulation

| constants      | Values   |
| ------------------ | ------------ |
| Speed of light $c$ | 299792458 m  |
| Carrier frequency $f_{B1I}$ | 1561.098 MHz |
| Ranging code length | 2046         |
| Ranging code rate $\bar f_{code}$ | 2.046 Mcps   |
| D1 Navigation message rate | 50 bps       |

## Signal Structure

The B1I signal is broadcast on medium Earth orbit (MEO), inclined geosynchronous orbit (IGSO), and geostationary orbit (GEO) satellites of BeiDou-2 and BeiDou-3, providing an open service. This signal modulates “ranging code plus navigation message” onto a carrier wave using binary phase shift keying (BPSK). Taking into account signal attenuation during propagation, its mathematical expression is as follows:

$$
\begin{equation}
S^j = P_c G^j(t) [c^j(t) N^j(t) D^j(t)]cos \phi^j + P_c G^j(t) [c^j(t) N^j(t) D^j(t)] sin \phi^j
\end{equation}
$$
In this equation, the superscript $j$ denotes the satellite number, $P_c$ represents the signal power, $G^{j}(t)$ is the signal gain at the receiver, $c(t)$ denotes the signal ranging code level at time $t$, $N(t)$ represents the NH code level of the secondary coding, and $D(t)$ denotes the data code level at the same time instant. The signal gain depends both on the propagation distance of the signal and on the elevation angle of the satellite; hence, it is time-varying. We denote the first term on the right-hand side of the equation as $I_j(t)$ (referred to as the in-phase component) and the second term as $Q_j(t)$ (referred to as the quadrature component).

## Ranging code (PRN)

The ranging code (also known as the spread-spectrum code) of B1I is generated by modulo-2 addition of two linear sequences, G1 and G2, followed by truncation of the last chip to produce a balanced Gold code. The correspondence between ranging code numbers and satellite types is as follows:

| 测距码编号 | 卫星类型      |
| ---------- | ------------- |
| 1~5        | GEO 卫星      |
| 6~58       | MEO/IGSO 卫星 |
| 59~63      | GEO 卫星      |

## Navigation message

According to their rates and structures, navigation messages are divided into D1 navigation messages and D2 navigation messages. The D1 navigation message has a rate of 50 bps and is modulated with a secondary code at a rate of 1 kbps. The navigation message contains basic navigation information (navigation parameters of this satellite), ephemeris data for all satellites, and time synchronization information with other GNSS systems. The D2 navigation message has a rate of 500 bps and includes basic navigation information as well as wide-area differential information (differential and integrity information for the BeiDou system, grid-point ionospheric information, etc.).

The B1I signal transmitted by BeiDou MEO/IGSO satellites uses the D1 navigation message, while the B1I signal transmitted by GEO satellites uses the D2 navigation message. In the first phase of the program, only the D1 navigation message is supported.

The D1 navigation message consists of superframes, master frames, and subframes. Each superframe contains 36,000 bits and takes 12 minutes to transmit in its entirety. Each superframe comprises 24 master frames (24 pages); each master frame contains 1,500 bits and lasts 30 seconds. Each master frame is further divided into 5 subframes; each subframe contains 300 bits and lasts 6 seconds. Each subframe consists of 10 words; each word contains 30 bits and lasts 0.6 seconds.

All words consist of two parts: the navigation message and the checksum. For the first word in each subframe, the first 15 bits carry no error-correction coding; the remaining 11 bits are encoded using the BCH(15,11,1) scheme, resulting in a total of 26 information bits. The other nine words all employ the BCH(15,11,1) scheme combined with interleaving for error-correction coding, yielding a total of 22 information bits.

D1 navigation messages broadcast basic navigation information in subframes 1 through 3 of each main frame. Subframes 4 and 5 are divided into 24 pages, which carry ephemeris data and time synchronization information with other GNSS systems. Therefore, during simulation, it is sufficient to generate only subframes 1 through 3 and pages 9 through 10 of subframe 5.

For detailed information on the structure of navigation messages, refer to the “BeiDou Satellite Navigation System Space Signal Interface Control Document.” The function for encoding RINEX file broadcast ephemerides into navigation messages has already been implemented in the program provided in Reference [1] (the `eph2sbf` function). However, this implementation couples message encoding with error-correction coding; therefore, these two processes need to be separated and handled uniformly during the stage of generating the actual broadcast navigation messages.

### Error-correcting coding

The navigation message data code is grouped in sequences of 11 bits each, and a 4-bit error-correction code is generated for each group. These 11-bit information segments, together with their corresponding 4-bit error-correction codes, form a BCH code that is 15 bits long. Every two groups of BCH codes undergo a parallel-to-serial conversion, bit by bit, to produce an interleaved code that is 30 bits long. In other words, each 30-bit interleaved code contains 22 bits of data code and 8 bits of error-correction code.

The functionality described in reference [1] has already been implemented in the provided program (in the functions `BCHcode` and `combBDSword`; see reference for details). The error-correcting decoding program has been implemented by the open-source library SDRLIB and can be used to test the error-correcting encoding functions.

### Secondary encoding

The secondary modulation of the D1 navigation message refers to modulating a Neumann-Hoffman code (referred to as the NH code) onto the D1 navigation message at a rate of 50 bps. The NH code has a period equal to the width of one navigation information bit (20 ms). The width of one bit in the NH code is the same as that of the ranging code (both being 1 ms), and the code rate is 1 kbps. The NH code is modulated synchronously with the ranging code and the navigation information code using modulo-2 addition. The 20-bit NH code is 00000100110101001110. The secondary modulation is illustrated in the figure below:

![二次编码示意图](assets/nh-code.png)

In the program provided in reference [1], secondary encoding is performed directly during the generation of ranging codes. The advantage of doing so is that the program logic is simple; however, the drawback is that the length of the ranging code is expanded to 2046 × 20 chips, consuming more memory.

## Satellite Position Calculation

The navigation information in the broadcast ephemeris can be used to compute the position of a satellite at any given time. The computation method is described in the “BeiDou Satellite Navigation System Space Signal Interface Control Document.” This functionality has been implemented by the open-source software RTKLIB and is reusable.

## Ionospheric Model

The BeiDou navigation message provides an 8-parameter Klobuchar model that can be used to compute the ionospheric vertical delay correction for the B1I signal:
$$
\begin{equation}
I_z(t) = \left\{
\begin{array}{lr}
5 \times 10^{-9} + A cos(2\pi \frac{t-50400}{T}), & |t-50400| < \frac T4 \\
5 \times 10^{-9}, & |t-50400| \ge \frac T4
\end{array}
\right.
\end{equation}
$$
In this equation, $t$ represents the local time at the piercing point M (with a value range of 0 to 86,400 seconds). The calculation formula is as follows:
$$
\begin{equation}
t = (t_E + \frac{43200 \lambda_m}{\pi})
\end{equation}
$$
Here, $t_E$ denotes the user’s measured time expressed in seconds within the week. $\lambda_m$ represents the geographic longitude of the ionospheric piercing point, with the unit being radians.

$A$ is the amplitude of the cosine curve representing daytime ionospheric delay, calculated using the coefficients $\alpha_n$:
$$
\begin{equation}
A = \left\{
\begin{array}{lr}
\sum_{n=0}^{3} \alpha_n |\frac{\phi_m}{\pi}|^n, & A > 0 \\
0, & A < 0
\end{array}
\right.
\end{equation}
$$
$T$ is calculated using the coefficients $\beta_n$:
$$
\begin{equation}
T = \left\{
\begin{array}{lr}
172800, & T \ge 172800 \\
\sum_{n=0}^3 \beta_n |\frac{\phi_m}{\pi}|^n, & 172800 \ge T \ge 72000 \\
72000, & T < 72000
\end{array}
\right.
\end{equation}
$$
In the above two equations, $\phi_m$ is the geographic latitude of the ionospheric pierce point, expressed in radians. The geographic longitude and latitude $λ_m$ and $\phi_m$ of the ionospheric pierce point are calculated as follows:
$$
\begin{equation}
\phi_m = arcsin(sin \phi_u cos \psi + cos \phi_u sin \psi cos Az)
\end{equation}
$$

$$
\begin{equation}
\lambda_m = \lambda_u + arcsin(\frac{sin \psi sin Az}{cos \phi_m})
\end{equation}
$$

In the above equation, $\phi_u$ represents the user’s geographic latitude, and $\lambda_u$ represents the user’s geographic longitude, both in radians; $Az$ denotes the satellite azimuth angle, also in radians; $\psi$ is the geocentric angular distance between the user and the pierce point, expressed in radians, and its calculation formula is as follows:
$$
\begin{equation}
\psi = \frac{\pi}{2} - El - arcsin(\frac{R}{R+h} cosEl)
\end{equation}
$$
Here, $R$ is the Earth’s radius, taken as 6378 km; $El$ is the satellite elevation angle, measured in radians; and $h$ is the height of the ionospheric single layer, set at 375 km.

In fact, the above calculation yields the zenith ionospheric delay, and it is also necessary to project $I_z$ onto the signal’s propagation direction:
$$
\begin{equation}
I = F I_z
\end{equation}
$$
Here, $F$ is the mapping function, which is computed as follows:
$$
\begin{equation}
F = \frac{1}{\sqrt{1 - (\frac{R}{R+h} cosEl)^2}}
\end{equation}
$$
The ionospheric delay $I$ calculated above is expressed in seconds. To convert it into distance, one must further multiply by the speed of light $c$. Unlike GPS, the BeiDou system broadcasts different ionospheric model parameters for each satellite (whereas GPS broadcasts the same ionospheric parameters for all satellites). The IGS recommends prioritizing the use of ionospheric parameters from GEO satellites, followed by IGSO satellites, and lastly MEO satellites.

## Signal Simulation Process

Before starting the simulation, the user is required to input a broadcast ephemeris file in RINEX format and set the receiver’s location, the duration of the simulated signal, the signal sampling rate, and other parameters. The signal simulation process is identical to that used for GPS signal simulation:

1. **Step 1:** Load and select a set of broadcast ephemerides. The RINEX file formats  containing BeiDou broadcast ephemerides are RINEX 3 and RINEX 4. The  file reading functionality is already implemented in RTKLIB and can be  reused.
2. **Step 2:** Calculate the satellite positions and determine the visible satellites  based on the receiver's position. Then, assign channels to the visible  satellites. The algorithms for this part have been described above.
3. **Step 3:** For the visible satellites, calculate the geometric distance  (satellite-to-ground distance) between the satellites and the receiver.  The algorithm for this part is relatively simple.
4. **Step 4:** For the visible satellites, calculate various signal errors, such as  satellite clock bias and ionospheric delay, and add them to the  satellite-to-ground distance to form the pseudorange.
5. **Step 5:** Calculate the signal gain based on the satellite's elevation angle and  the satellite-to-ground distance. This part can directly refer to the  GPS implementation.
6. **Step 6:** Calculate the ranging code, NH code, and data code phases of the  simulated signal based on the pseudorange. This part is more complex and needs to be broken down into the following processes:
   1. Generate the ranging code based on the satellite's PRN number. The algorithm for this part has been described above, and reusable code is available.
   2. Generate the data code based on the broadcast ephemeris. This requires  converting the navigation information into raw bits on subframes, and  then obtaining the interleaved code through error-correction encoding.
   3. Calculate the phases of the ranging code, NH code, data code, and carrier wave  based on the pseudorange. The generation process will be described  below.
7. **Step 7:** Calculate the signal I/Q values based on the ranging code, NH code, data code phase, carrier phase, and signal gain.
8. **Step 8:** Accumulate the I/Q values of all visible satellites to obtain the final signal.

It should be noted that the broadcast ephemeris of BeiDou may be updated at the beginning of each superframe; therefore, you need to check whether the broadcast ephemeris should be updated at the 0th, 12th, 24th, 36th, and 48th minutes of each hour (BDT).

## Signal Phase Calculation

As shown in Formula (1), the core issue in GNSS signal simulation is the computation of ranging codes, NH codes, data codes, and carrier phase. This section derives the corresponding computational methods.

### Ranging code, NH code, and data code phase

A B1I ranging code consists of 2,046 chips, with a ranging code rate of 2.046 Mcps. Consequently, the ranging code period is 1 ms. The D1 navigation message has a transmission rate of 50 bps; each word comprises 30 bits, and every 10 words form a subframe. Therefore, the duration of a data-code bit is 20 ms, while the duration of a word is 600 ms. To facilitate time synchronization between the satellite and the receiver, the navigation message includes the start time of each subframe. If we imagine the subframe as a ruler, then the words, bits, ranging codes, and chips of the data code can all be regarded as scale markings on this ruler. Thus, the pseudorange ρ measured by the receiver can be expressed by the following equation:
$$
\begin{equation}
\rho = \frac c{1000}(20n_b + n_c + \frac{n_{cp}}{2046}),
\end{equation}
$$
where $n_b$ and $n_c$ represent the number of words and bits, respectively, contained in the pseudorange, and $n_{cp}$ denotes the number of chips that fall short of one full pseudocode period (i.e., the pseudocode phase).

To facilitate the calculation of which “word” the receiver is tracking at a given moment, we can shift the time reference back to the beginning edge of the frame. We define four symbols: $t_r$, $t_f$, $t_p$, and $t_t$, representing respectively the time when the receiver receives the signal, the time when the frame begins, the signal transmission time, and the signal transmission time, all measured in milliseconds. Then:
$$
t_t = t_r - t_p, \\
t_p = 1000 \rho / c.
$$

The calculation method for $n_w$ is as follows:
$$
\begin{equation}
n_w = \lfloor (t_t - t_f) / 600 \rfloor = \lfloor (t_r - t_f - t_p) / 600 \rfloor，
\end{equation}
$$
The parameters in Equation (5) can be calculated using the following formula:
$$
\begin{equation}
n_b = \lfloor (t_r - t_f - t_p)/20 - 30n_w \rfloor，
\end{equation}
$$

$$
\begin{equation}
n_c = \lfloor (t_r - t_f - t_p) - 600n_w - 20n_b \rfloor，
\end{equation}
$$

$$
\begin{equation}
n_{cp} = \lfloor 2046 [(t_r - t_f - t_p) - 600n_w - 20n_b - n_c] \rfloor，
\end{equation}
$$

In the formula, the symbol $\lfloor \rfloor$ denotes the floor function. To obtain the ranging code level $c(t)$, we take the $n_{cp}$th state from the satellite ranging code sequence; to obtain the NH code level $N(t)$, we take the $n_c$th bit from the NH code sequence; and to obtain the data code level $D(t)$, we take the $n_b$th bit of the $n_w$th word from the satellite navigation message. It is important to note that the time difference $t_r - t_f - t_p$ calculated in the above formula may be negative! Therefore, it is advisable to shift the time reference back to just before the start of the 5th subframe of the previous frame—i.e., by adding 6 seconds to $t_r - t_f - t_p$. The resulting value of $n_w$ thus obtained corresponds to the $n_w$th word starting from the 5th subframe of the previous frame.

If you need to add error terms (such as satellite clock errors, ionospheric delays, etc.) to the simulated pseudorange measurements, you can convert these errors into distances and directly add them to $\rho$.

### Carrier phase

The initial value of the carrier phase can be referenced from the GPS-SDR-SIM software’s handling of GPS carrier phases (or set it to zero):
$$
\begin{equation}
\phi = \frac{(2 \rho_o - \rho)}{\lambda_{B1I}}
\end{equation}
$$
In the above equation, $\lambda_{B1I}$ represents the wavelength of the B1I carrier, and $\phi$ is a floating-point number. To obtain the carrier phase that is less than one full cycle, the following formula can be used:
$$
\begin{equation}
\hat \phi = \phi - \lfloor \phi_i \rfloor
\end{equation}
$$

### Calculation of Signal Frequency

Due to the relative motion between the satellite and the receiver, the Doppler effect causes a frequency shift in both the carrier wave and the pseudocode. The impact of the Doppler effect can be calculated using the following equation:
$$
\begin{equation}
\dot f_{carr} = - \frac{\rho_i - \rho_{i-1}}{\lambda_{B1I} T}
\end{equation}
$$

$$
\begin{equation}
\dot f_{code} = \bar f_{code} + \frac{\bar f_{code}}{f_{B1I}} \dot f_{carr} = \bar f_{code} + \frac {\dot f_{carr}}{763}
\end{equation}
$$

In these equations, $\rho_i$ and $\rho_{i-1}$ represent the pseudoranges at adjacent time instants $T$ ($T = 0.1,\text{s}$), $\dot f_{carr}$ and $\dot f_{code}$ are the carrier frequency and the ranging-code frequency, respectively, and $\bar f_{code}$ is the average ranging-code frequency (2.046 Mcps). In practice, Equations (12) through (17) are used only to compute the initial values of the pseudocode phase and the carrier phase; the pseudocode phase and the carrier phase at subsequent sampling points can be calculated using the following formula:
$$
\begin{equation}
\hat \phi_{i+1} = \hat \phi_i + \delta \phi_i = \hat \phi_i + \Delta t \dot f_{carr}
\end{equation}
$$

$$
\begin{equation}
n_{cp(i+1)} = n_{cp(i)} + \delta n_{cp}  = n_{cp(i)} + \Delta t \dot f_{code}
\end{equation}
$$

Here, $\Delta t$ represents the signal sampling interval, while $\delta \phi_i$ and $\delta n_{cp}$ are referred to as the carrier phase step and the ranging code step, respectively.

### Signal Gain Calculation

The signal gain depends on two factors: first, the distance over which the signal propagates; and second, the elevation angle at which the signal impinges. The total antenna gain is decomposed into two components:
$$
\begin{equation}
G^{j} = G^{j}_d G^{j}_e
\end{equation}
$$
In this equation, $G^{j}d$ represents the component related to distance, while $G^{j}e$ represents the component related to the elevation angle. Referring to GPS-SDR-SIM, $G^{j}e$ is described using a grid model, so only the distance-related component needs to be calculated:
$$
\begin{equation}
G^{j}_d = \frac {H_{sat}}{D_{sr}}
\end{equation}
$$
Here, $H{sat}$ denotes the satellite's vertical height above the Earth's surface (for MEO satellites, it is 21,528 km; for GEO/IGSO satellites, it is 35,786 km), and $D{sr}$ represents the distance from the satellite to the receiver.

## Other

### RINEX Format Compatibility

Only RINEX versions 3 and above support multi-system broadcast ephemerides. Therefore, it is essential to achieve compatibility with higher-version RINEX formats. 

### GPST 与 BDST 的转换

For future program expansion, the program internally uses GPST uniformly to represent time. This requires converting the time (BDT) from the BeiDou broadcast ephemeris. Both GPST and BDT represent time using a week count (Week Number, WN) and a second of the week (Second of Week, SOW); the key difference lies in:

- The start time (or epoch) of GPST is 00:00 on January 6, 1980, and the start time of BDT is 00:00 on January 1, 2006.
- The difference between GPST and UTC is the number of leap seconds since the start of GPST, and the difference between BDT and UTC is the number of  leap seconds since the start of BDT.

Therefore, the conversion relationship between the two is:
$$
\begin{equation}
t_{BDT} = t_{GPST} - 14 + \Delta t_{BDT-GPST}
\end{equation}
$$
In this equation, $t_{BDT}$ and $t_{GPST}$ represent BDT and GPST, respectively, with units in seconds; 14 is the fixed difference between the two systems; and $\Delta t_{BDT-GPST}$ represents the small time-system difference between the two systems.
